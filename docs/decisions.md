# 需求取捨決策

> 這份文件可以由 Codex 先產生草稿，但最後必須由你刪除、修改或補充。請不要讓 AI 替你做最後決策。

## 我決定 v1 優先服務的使用者

- [ ] 回報者
- [ ] 資訊整理者
- [x] 行動者

原因：人類決定 v1 先服務「行動者」。Phase 0 的觀察顯示，很多原始資訊看起來像可以派工，但其實還缺少位置、時間、當事人確認、現場狀態或風險判斷。若行動者誤把未確認資訊當成可執行任務，後果比畫面不好用更嚴重。

## 這個使用者最想完成什麼

- 快速看出哪些資訊目前仍未確認，哪些資訊不適合直接採取行動。
- 在準備下一步前，看見原文、查核狀態、卡住原因與需要人工確認的地方。

## 這個使用者最怕什麼錯誤

- 把未確認資訊當成已確認任務。
- 根據模糊、過期或互相衝突的資訊跑錯地方、帶錯物資，或做出不安全行動。
- AI 把「可能」「有人說」「轉述」整理成看起來很確定的指示。

## 我今天暫時不處理的需求

- 暫時不優先做回報者的快速填寫流程。
- 暫時不把整理工作台擴成正式派工系統或任務管理系統。
- 暫時不加入地圖、外部查證、後端資料庫或即時通知。

## 我不採用的 AI 建議

- AI 建議：把比較完整的原始資訊直接整理成「可行動任務」或「優先任務」。
  - 不採用原因：Phase 0 資料仍是原始資訊，完整度較高不等於已確認；v1 服務行動者時，應先避免誤導行動，而不是追求自動派工。

## 必須保留人工確認的判斷

- 資訊是否已確認。
- 是否可以從原始資訊建立行動建議。
- 模糊地點、當事人同意、需求是否仍存在、資訊是否過期或衝突。
- AI 候選整理是否補了原文沒有明確說的內容。

## 今天明確不做什麼

- 不做真實救災判斷。
- 不把 `needs_review` 或 `unverified` 顯示成已確認。
- 不把 v1 做成正式派工、路線規劃或救災決策工具。
- 不新增後端、資料庫、外部 API、地圖或 runtime LLM API。

## 我仍不確定的地方

- 行動者需要看到多少原文細節，才足以避免誤判，但又不會被資訊量淹沒。
- v1 應該用什麼文字描述「不可直接行動」，才不會讓使用者誤以為只是低優先度。
- 哪些人工確認欄位應該成為必填，哪些只要以提醒方式呈現。
