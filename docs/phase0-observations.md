# Phase 0 Observations

## 我們做出了什麼？

- 首頁可以看到第一階段原始資訊列表，並能切換到整理工作台。
- 整理工作台目前只使用 `src/fixtures/phase-0/messy-reports.json`，沒有新增後端、資料庫、外部 API 或真實資料。
- 工作台先建立 6 筆可編輯整理草稿，讓小組可以修改候選類型、信心程度、原文依據、卡住原因、人工確認筆記，並可刪除或重設草稿。

## 哪些資料讓 agent 很容易整理？

- M-010 文字中有時間、數量、物資狀態與下一次盤點時間，比較容易先整理成「地點狀態候選」。但它仍是 `needs_review`，不能顯示成已確認。
- M-009 有明確回報時間、地點方向與限制條件，適合先整理出「哪些內容需要同步確認」，但不能直接發布為正式集合點規則。
- M-003 有新舊資訊衝突的線索，容易看出「原本那張單可能沒更新」，適合標示為需要人工確認。

## 哪些資料讓 agent 亂補或誤判？

- M-001 可能讓 agent 自動把「老雜貨店後面」想成可派工地點，但原文沒有完整地址，也沒有確認需求人數與當事人。
- M-005 的 `sourceType` 是 `official_notice`，但原文說不知道截圖日期與是否官方公告；不能因欄位名稱就當作已確認公告。
- M-012 容易被 agent 直接轉成藥品任務，但原文只有外地家屬轉述，且無法確認親友目前位置。

## 哪些資訊不能直接變成任務？

- M-001：只有模糊位置與「有人說」，不能直接派十幾個人。
- M-006：集合點資訊彼此衝突，可能有淹水風險。
- M-008：只說 A 區先不要再派人，但不知道原因是人太多、道路危險或任務完成。
- M-011：牽涉長者住家與完整地址公開同意，不能直接派人到場。
- M-012：需求、位置與當事人狀態都尚未確認。

## 哪些資訊的操作者不是當事人？

- M-011 是現場志工代長者轉述，還沒有確認長者是否同意公開完整地址。
- M-012 是外地家屬來電，不在現場，也無法確認親友位置。
- M-004 是群組轉述「有人說」，不是現場值守者或當事人直接確認。

## 我們剛剛偷偷假設了什麼？

- 假設出現時間與數量就代表比較可靠，但來源仍可能需要查核。
- 假設「現場志工」一定知道完整現況，但志工也可能只看到局部資訊。
- 假設資訊可以直接變成任務；其實還需要同意、位置、風險、資源狀態與負責確認者。

## 如果要讓這個工具變可靠，下一步需要什麼？

- 先定義整理後資料格式，清楚區分原文、候選判斷、人工確認結果與正式任務。
- 補上人工確認角色與責任，例如誰能確認來源、誰能確認位置、誰能決定是否建立任務。
- 加入狀態流程，避免從 `needs_review` 直接跳成已確認或可派工。

## 我們需要更清楚資料格式的地方

- 來源類型與查核狀態要分開，`sourceType` 不能取代 `verificationStatus`。
- 候選類型、原文依據、推測內容、卡住原因要分欄，不要混在同一段文字。
- 需要能記錄操作者是否為當事人、是否有授權、是否能公開地址或聯絡方式。

## 我們需要人工確認的地方

- M-005 的截圖日期與是否為官方公告。
- M-006 的集合點是否安全，哪一則現場回報較新。
- M-011 的當事人同意、位置描述與是否適合建立任務。
- M-012 的親友位置、藥品需求是否存在，以及是否應建立任務。

## 60 秒分享草稿

我們做了一個 Phase 0 前端整理工作台，讓原始資訊、查核狀態、候選草稿與卡住原因同時可見。這個成果最有用的地方不是自動分類，而是提醒我們：很多資料看起來像任務，實際上仍缺來源、時間、地點、當事人同意或現場確認。AI 最容易誤導我們的地方，是把模糊轉述整理成好像可以直接行動的任務；所以工作台把草稿維持為候選，並要求人工確認。
